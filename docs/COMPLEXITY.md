# Accepted complexity exceptions

This file is the single source of truth for accepted cyclomatic-complexity
exceptions in the shipped `waitbus/` package. It is verified by
`tests/test_complexity_table.py`, which asserts the table here exactly equals
`radon cc waitbus -s -n C` output — any untabled C+ function or any
stale/phantom row fails the test. The `scripts/` and `benchmarks/` trees are
each held by a separate D+ ratchet in the same test.

Methods at cyclomatic-complexity grade C (complexity 11 or higher) are acceptable when the complexity is inherent to the problem — orchestration of operator-visible steps, algorithmic necessity, fault isolation across independent sections, type dispatch on a discriminator, or traversal of irreducibly tree-shaped data — rather than organisational. Refactoring in those cases typically produces single-use indirections that hurt readability without reducing real cognitive load. Each accepted function is listed below with its exception class.

**Scope: the C+ table ratchet covers `waitbus/` only — `benchmarks/` and `tests/` are excluded from this C+ table; `scripts/` and `benchmarks/` are each held by a separate D+ ratchet.** Bench scripts, soak harnesses, and orchestration test scenarios are top-of-the-funnel multi-step orchestrators by nature (every bench is "set up host, spawn daemon, drive workload, sample, write artefact"). Forcing each to A(5) would create exactly the single-use-indirection anti-pattern the orchestration exception class above accepts; documenting each as an "orchestration" exception in this table would be paperwork without signal. The shipped library is where complexity discipline materially affects long-term maintenance, and the C+ ratchet is scoped accordingly.

**`scripts/` second-tier ratchet at D+ (cyclomatic >= 21).** Added 2026-05-22 in response to the audit's CRIT-02 finding that `scripts/soak.py::main` jumped from D(22) to E(37) in one session — the exact "bench grows to operator-visible C(20)+" trigger the prior version of this DEC anticipated. The threshold is D rather than C so the ratchet does not flag the inherent orchestration complexity `scripts/` legitimately carries (multi-stage automation, parameter-sweep loops, validation rule-groups all live at C-grade) and only fires when a script crosses the operator-visible line the prior DEC named verbatim. Verified by `tests/test_complexity_table.py::test_scripts_dir_has_no_d_or_worse_functions`, which asserts `radon cc scripts -s -n D` returns the empty set; any new D-grade function in `scripts/` requires either a refactor below the threshold or an explicit DEC entry documenting why the refactor is wrong. If a future audit needs the ratchet extended further (e.g. to `benchmarks/`), extend the test's `radon cc` invocation rather than carving exceptions in this table.

**`benchmarks/` second-tier ratchet at D+ (cyclomatic >= 21).** Added 2026-06-04 in response to the pre-launch audit finding that the measurement suite sat outside both quality forcing-functions (coverage source is `waitbus/` only; the complexity ratchet covered `waitbus/` + `scripts/`). That blindness was the structural reason the audit's findings concentrated in `benchmarks/`: `bench_multistream_proof.py::_build_verdict` had accreted to F(118) -- six times the operator-visible C(20) line -- and `bench_polling_vs_subscribe_llm_agent.py::_run_one_iteration` regrew F(30) -> F(54) since the prior audit, with no gate to catch either. This is exactly the extension the policy below anticipated, so it is the empty-set D+ form -- no per-function exemption dict -- matching the `scripts/` tier. Arming it required first refactoring the four D+ bench orchestrators below the threshold: `_build_verdict` F(118) -> C(20) (table-driven substrate registry), `_run_bench` E(31) -> B(8), `_run_one_iteration` F(54) -> C(14) and the polling `main` E(34) -> C(13), and the streaming `main` F(43) -> thin orchestrator. Verified by `tests/test_complexity_table.py::test_benchmarks_dir_has_no_d_or_worse_functions`, which asserts `radon cc benchmarks -s -n D` returns the empty set; any new D-grade bench function requires a refactor below the threshold or an explicit DEC entry documenting why the refactor is wrong.

**Coverage: `benchmarks/` stays out of the line-coverage source set, compensated by contract tests.** The audit's "bring benchmarks/ under coverage" finding is addressed by a documented exemption rather than by forcing the 80%/file gate. `benchmarks/` is execution code -- it spawns the real daemon and drives real LLM subprocesses -- so its lines execute only under a live, keyed, host-pinned run; a unit-coverage target would be reachable only by mocking away the very thing each bench measures, and the suite is excluded from the sdist (it is not shipped surface). In place of line coverage, `benchmarks/` is held by four compensating contracts: (1) `mypy --strict` over the whole tree; (2) this D+ complexity ratchet; (3) the harness-helper-adoption contract (`tests/test_bench_helper_adoption.py`); and (4) per-bench property/boundary tests that pin each verdict's pure aggregation logic against synthetic inputs (`tests/test_bench_multistream_proof.py`, `tests/test_bench_polling_vs_subscribe_llm_agent.py`, `tests/test_bench_streaming_reasoning_integrity.py`). These gate the verdict math and the structural conventions without requiring a live measurement run in CI.

| `wait.py::_build_predicate` | C(19) | orchestration | Lowers the CLI's `--sha` / `--match` / `--cond` / `--match-cel` / `--match-jmespath` inputs into one canonical AND-composed predicate. Each branch is an operator-visible input shape with a distinct startup-2 error contract (sugar conflict, empty sha, the `--sha` hex/length validation surfaced by `head_sha_prefix`, malformed match, missing evaluator extra, no predicate supplied) -- collapsing them through a generic loop would lose the per-arm remediation string. `_wait` itself is now A(4) by delegating to this helper plus `_resolve_repo` (B 9) and `_dispatch_outcome` (B 7). |
| `broadcast.py::Broadcast._fan_out` | C(12) | orchestration + fault isolation | Per-subscriber fan-out loop with four independent concerns: filter match, replay-watermark dedup (skip + clear-on-first-live), pre-ack buffer gate (queue + bounds + overflow-into-drop), and live send + lag-count drop. Each concern has a distinct per-subscriber side effect (cursor advance vs buffer append vs sendall vs drop) and metrics emission; collapsing them through helpers would force a per-iteration outcome tuple that re-dispatches inside the loop, breaking the compute→record→advance locality the existing shape gives. Lands as part of the policy below (the pre-ack gate is the new arm; the prior C-grade was B). |
| `broadcast.py::Broadcast._read_subscribe` | C(17) | orchestration | Subscribe handshake orchestrator: parse + validate envelope → proto-version reject → optional token reject → field-validator dispatch → register subscriber → reserve pre-ack buffer → compute caught_up_at → emit subscribe_ack → optionally replay (guarded against a mid-replay `sqlite3.Error` → silent close, and an in-replay lag eviction → early return) → drain pre-ack buffer → clear buffer. Each stage has a distinct externally-observable outcome (silent EOF on shape error, typed `subscribe_rejected` frame on auth/version failure, full success path) and must run in strict sequence (the buffer reservation has to precede registration; the ack must precede replay; the drain must follow replay). Splitting into sub-orchestrators creates single-use indirections that hurt readability and obscure the ordering invariant the the policy below amendment encodes. (Rose from C(16) to C(17) with the replay DB-error guard.) |
| `_broadcast_sub.py::await_predicate` | C(16) | fault isolation | Independent terminal/skip arms: timeout / framing-EOF / clean-EOF / undecodable-skip / control-frame-drain (`daemon_heartbeat` + `subscribe_ack` are consumed off the wire but never handed to `decide` or advanced onto the cursor) / subscribe-rejected → typed exception via the `_REJECT_REASON_EXCEPTIONS` map (the policy below) / match / SIGINT. Each is a distinct externally-observable contract; decomposing forces tuple-return + re-dispatch and breaks compute→tag locality. (Dropped from C(18) to C(17) when the if/else reason dispatch collapsed to a single map lookup, then to C(16) when the reject decode-and-raise moved into the shared `_raise_for_reject` helper.) |
| `sources/docker_watch.py::watch` | C(14) | fault isolation | Container event stream loop: connect → stream → reconnect/error arms. The capped backoff, event-epoch precedence and exit-code coercion are extracted to single-purpose helpers; the residual in-function arms are the fault-isolation core (stream-yielded / reconnect / unrecoverable) whose per-arm recovery contract does not decompose further without re-coupling them through a tuple-return. (Rose C(12)->C(14) when the cooperative stop seam landed: the stop-flag checks at the loop top and in the drop handler are per-arm shutdown contracts, same fault-isolation class.) |
| `sources/docker_watch.py::_iter_event_lines` | C(11) | fault isolation | Chunked /events reader: connect, register-with-stopper, per-line yield, and the generator finally that owns the connection close. The branches are the stop-seam contract (stopper registered around the live connection so a cross-thread shutdown wakes the blocked read; the reading thread alone closes) plus the HTTP-status and short-read arms — each a distinct recovery path that consolidating would re-entangle with the watch loop's own arms. |
| `_terminal.py::entity_key` | C(12) | type dispatch | Per-source / per-event-type dispatch (github/workflow_run, github/workflow_job, alertmanager/prometheus_alert, plus the pass-through default for sources without a stable upstream identity) with `type(v) is int and v > 0` domain narrowing on each GitHub id column and `isinstance(str)` plus non-empty narrowing on the alertmanager fingerprint. Each branch encodes a per-source identity contract that the coalesced consumer collapses on; collapsing them through one generic accessor would lose the per-source error semantics (e.g. silently coercing `bool` to int-stringified `True`/`False` collapse keys). |
| `cli/_shared.py::_render_table` | C(12) | type/value dispatch | Fixed-width aligned-table renderer shared by ``waitbus source list`` and ``waitbus allowlist list``. Per-column width compute, per-row stringify, header bolding when colour is enabled, optional per-row colour callback (plugin rows in ``source list`` go CYAN; allowlist rows are uncoloured). Each branch handles one presentation concern; splitting would force the caller to pass a pre-bolded header AND a pre-formatted row sequence, losing the centralisation that closes the previous duplication. |
| `cli/sources.py::source_show` | C(13) | orchestration | The 12-field deep-dive verb for one source: registry lookup, plugin-vs-built-in branch, optional PEP 740 attestation JSON read off disk (via the public `read_attestation_json` helper), TOFU pin lookup, optional `--json` format, payload-schema fully-qualified-name render. Each branch contributes one operator-visible field; per-field helpers would be single-use. The same `source list` row helper is reused for the table cells. |
| `sources/_attestation.py::verify_distribution` | C(11) | fault isolation + type dispatch | PEP 740 verification flow: locate the dist-info, read the PEP 610 `direct_url.json` for the wheel digest (graceful None on pip-name installs that don't write it), iterate `Provenance.attestation_bundles`, call `Attestation.verify` against each bundle's publisher identity. Each arm has a distinct fall-soft / fall-hard contract -- missing dist-info / missing direct_url.json / missing provenance file gracefully return None so the daemon can continue; sigstore `VerificationError` / `ConversionError` / malformed envelopes raise `AttestationVerificationError` so the registry's TOFU policy treats the plugin as untrusted. Splitting the arms into helpers loses the per-arm log-then-continue + raise-then-stop discrimination that the surrounding `_verify_publisher` relies on. |
| `sources/_protocol.py::SourceSpec.__post_init__` | C(12) | type/value dispatch | Per-field validation across the four `SourceSpec` fields (`name` regex, `event_types` non-empty + per-member regex, `payload_schema` isinstance, `api_version` positive-non-bool). Each branch raises a field-specific `ValueError` so plugin authors see which field failed; collapsing into a single error path would lose the per-field message and force authors to debug by inspection. The msgspec.Struct construction path requires this complexity because plugin authors are not in waitbus's type-check scope, so the runtime check is the only enforcement seam. |
| `sources/_config.py::load_plugin_policy` | C(12) | fault isolation | Three independent sources merged: `config.toml [plugins]` table (with per-field type-narrow guards on `autoload` / `allow` / `deny`), the `WAITBUS_DISABLE_SOURCE_AUTOLOAD` env override, and the `WAITBUS_PLUGINS` env list. The TOML parse + missing-file fallback + each guard arm produces a typed `PluginPolicy` regardless of which inputs are present; the per-field guards exist because TOML parse can yield wrong-type values when a hand-edited file is malformed. Splitting forces tuple-return through three private helpers that re-merge -- worse than the local accumulation. |

### Retired entries

Kept here as historical markers so a `git log -G "cli.py::main"` on this file surfaces the transition:

- `cli.py::main` — previously listed as C+; extracted to A(1) in commit `9424ded` (single-CLI-entry collapse). The dispatch complexity moved into the per-callback `_root` functions and the leaf commands above. The old entry's "type dispatch" rationale still describes typer's per-branch help generation — but the named function no longer carries the complexity.
- `cli.py::doctor` — previously listed as C+; extracted to A(5) (the originally-cited commit hash was a typo and is not uniquely recoverable; the A(5) `_check_*` decomposition is verifiable in the current `doctor` source). The fault-isolation rationale still applies to the per-section `_check_*` helpers (`_check_launchd` is in the table; `_check_systemd` is below the C+ threshold post-refactor).
- `cli.py::keygen` — previously listed as C(12); function deleted alongside the systemd-creds swap. The replacement `install-credentials` is a single-credential operation with no per-service loop, currently A(3).
- `cli.py::*` family — `cli.py` was split into the `cli/` package (zero surface change); the surviving C+ functions are re-keyed above to their new module paths (`cli/install/systemd.py`, `cli/install/launchd.py`, `cli/_shared.py`, `cli/status.py`).
- `etag_poll.py::main` — previously listed as C(12) orchestration; extracted to A(5) when the per-repo / per-run / per-batch loop bodies moved into `_poll_one_repo`. The orchestration is now a thin driver below the C+ threshold.
- `watchdog_check.py::main` — previously listed as C(14); the outcome state machine moved into `_check_once`, which is the table entry above. `main` is now a slim argparse wrapper.
- `replay.py::_replay` — previously D(21); split into `_resolve_cursor` + `_handle_frame` + `_stream_until_idle`. `_replay` is now an A(5) dispatcher and none of the helpers reach the C+ threshold, so there is no table entry.
- `read_events.py::watch` — previously listed as C(15) fault isolation; converged onto the shared `await_predicate` subscriber engine in commit `f342b90`, dropping the bespoke frame-loop. `watch` is now A(5) and below the C+ threshold, so there is no table entry; `_broadcast_sub.py::await_predicate` (in the table above) now carries the fault-isolation complexity.
- `broadcast_tap.py::_tap` — previously listed as C(14) fault isolation; converged onto the shared `await_predicate` subscriber engine in commit `f342b90`, dropping the bespoke receive-loop. `_tap` is now B(8) and below the C+ threshold, so there is no table entry; the fault-isolation complexity moved into `_broadcast_sub.py::await_predicate`.
- `_broadcast_sub.py::open_subscriber` — previously listed as C(15) fault isolation; the wire-protocol-v1 freeze (the policy below) removed the synchronous token-reject probe and the `prefetched` carry-over (the daemon's `subscribe_ack` / `subscribe_rejected` is now recognised by `await_predicate`), so `open_subscriber` dropped below the C+ threshold. No table entry; the reject-handling complexity consolidated into `await_predicate` (above).
- `cli/db/migrate.py::migrate` — previously C(15); split into `_print_status` + `_print_dry_run` + `_apply`. `migrate` is now an A(4) verb dispatcher below the C+ threshold.
- `pr_monitor.py::main` — previously C(20); the tick-loop body moved into `_pr_monitor_tick` (B(10)) and owner/repo resolution moved into `_resolve_owner_repo` (A(4)). `main` is now B(10) and below the C+ threshold, so there is no table entry.
- `cli/status.py::status` — previously C(19); decomposed into `_status_db` (B(7)), `_status_liveness` (C(11)), and `_status_socket` (A(2)). `status` is now an A(2) thin dispatcher. `_status_liveness` retains C(11) complexity (documented above) because the `match sys.platform` type-dispatch and per-daemon fault-isolation loop are inherent to the section's responsibility.
- `broadcast.py` `import asyncio as asyncio` rebind — previously held in place so tests could `monkeypatch.setattr(broadcast.asyncio, "run", ...)`. Dropped (per greenfield-no-alias policy): `mock.patch("waitbus.broadcast.asyncio.run", ...)` resolves the attribute path via `getattr` on the imported module regardless of whether the `as` re-export is present, so the rebind was load-non-bearing. Tests migrated to full-path `mock.patch` in the same commit.
