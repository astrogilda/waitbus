# Accepted complexity exceptions

This file is the single source of truth for accepted cyclomatic-complexity
exceptions in the shipped `waitbus/` package. It is verified by
`tests/test_complexity_table.py`, which asserts the table here exactly equals
`radon cc waitbus -s -n C` output — any untabled C+ function or any
stale/phantom row fails the test. The `scripts/` and `benchmarks/` trees are
each held by a separate D+ ratchet in the same test.

Methods at cyclomatic-complexity grade C (complexity 11 or higher) are acceptable when the complexity is inherent to the problem — orchestration of operator-visible steps, algorithmic necessity, fault isolation across independent sections, type dispatch on a discriminator, or traversal of irreducibly tree-shaped data — rather than organisational. Refactoring in those cases typically produces single-use indirections that hurt readability without reducing real cognitive load. Each accepted function is listed below with its exception class.

**Scope: the C+ table ratchet covers `waitbus/` only — `benchmarks/` and `tests/` are excluded from this C+ table; `scripts/` and `benchmarks/` are each held by a separate D+ ratchet.** Bench scripts, soak harnesses, and orchestration test scenarios are top-of-the-funnel multi-step orchestrators by nature (every bench is "set up host, spawn daemon, drive workload, sample, write artefact"). Forcing each to A(5) would create exactly the single-use-indirection anti-pattern the orchestration exception class above accepts; documenting each as an "orchestration" exception in this table would be paperwork without signal. The shipped library is where complexity discipline materially affects long-term maintenance, and the C+ ratchet is scoped accordingly.

**`scripts/` second-tier ratchet at D+ (cyclomatic >= 21).** The threshold is D rather than C so the ratchet does not flag the inherent orchestration complexity `scripts/` legitimately carries (multi-stage automation, parameter-sweep loops, validation rule-groups all live at C-grade) and only fires when a script crosses into the D-grade range. Verified by `tests/test_complexity_table.py::test_scripts_dir_has_no_d_or_worse_functions`, which asserts `radon cc scripts -s -n D` returns the empty set; any new D-grade function in `scripts/` requires either a refactor below the threshold or an entry in this table documenting why the refactor is wrong. If the ratchet needs extending further (e.g. to `benchmarks/`), extend the test's `radon cc` invocation rather than carving exceptions in this table.

**`benchmarks/` second-tier ratchet at D+ (cyclomatic >= 21).** This is the empty-set D+ form — no per-function exemption dict — matching the `scripts/` tier. Arming it required first refactoring the four D+ bench orchestrators below the threshold: `_build_verdict` F(118) -> C(20) (table-driven substrate registry), `_run_bench` E(31) -> B(8), `_run_one_iteration` F(54) -> C(14) and the polling `main` E(34) -> C(13), and the streaming `main` F(43) -> thin orchestrator. Verified by `tests/test_complexity_table.py::test_benchmarks_dir_has_no_d_or_worse_functions`, which asserts `radon cc benchmarks -s -n D` returns the empty set; any new D-grade bench function requires a refactor below the threshold or an entry in this table documenting why the refactor is wrong.

**Coverage: `benchmarks/` stays out of the line-coverage source set, compensated by contract tests.** benchmarks/ stays out of the line-coverage source set, compensated by contract tests rather than by forcing the 80%/file gate. `benchmarks/` is execution code -- it spawns the real daemon and drives real LLM subprocesses -- so its lines execute only under a live, keyed, host-pinned run; a unit-coverage target would be reachable only by mocking away the very thing each bench measures, and the suite is excluded from the sdist (it is not shipped surface). In place of line coverage, `benchmarks/` is held by four compensating contracts: (1) `mypy --strict` over the whole tree; (2) this D+ complexity ratchet; (3) the harness-helper-adoption contract (`tests/test_bench_helper_adoption.py`); and (4) per-bench property/boundary tests that pin each verdict's pure aggregation logic against synthetic inputs (`tests/test_bench_multistream_proof.py`, `tests/test_bench_polling_vs_subscribe_llm_agent.py`, `tests/test_bench_streaming_reasoning_integrity.py`). These gate the verdict math and the structural conventions without requiring a live measurement run in CI.

| `mcp.py::_tool_get_pr_aggregate_impl` | C(13) | nested data traversal | Walks the PR-aggregate query result into the tool's structured payload, with per-section presence guards across the runs / jobs / checks shapes; the branching mirrors the optional-field structure of the aggregate row. |

| `_db.py::_run_migration_in_transaction` | C(15) | algorithmic | SQLite `ALTER TABLE` limits force a rename pass + ADD COLUMN pass + DDL replay + SQLITE_BUSY retry in one transaction; the migration pattern doesn't decompose without losing transactionality. |
| `etag_poll.py::emit_stall_synthetic_events` | C(14) | fault isolation | SELECT candidates → per-candidate parse + filter → BEGIN IMMEDIATE batch → per-row insert + log → commit/rollback; dense but cohesive (one operation: detect-stalls-and-emit). |
| `etag_poll.py::_do_conditional_get` | C(12) | fault isolation | Conditional-GET with ETag revalidation: 304 / 200 / rate-limit / transient-error / hard-error arms each have distinct cache-write and retry semantics. |
| `wait.py::_build_predicate` | C(19) | orchestration | Lowers the CLI's `--sha` / `--match` / `--cond` / `--match-cel` / `--match-jmespath` inputs into one canonical AND-composed predicate. Each branch is an operator-visible input shape with a distinct startup-2 error contract (sugar conflict, empty sha, the `--sha` hex/length validation surfaced by `head_sha_prefix`, malformed match, missing evaluator extra, no predicate supplied) -- collapsing them through a generic loop would lose the per-arm remediation string. `_wait` itself is now A(4) by delegating to this helper plus `_resolve_repo` (B 9) and `_dispatch_outcome` (B 7). |
| `broadcast.py::Broadcast._fan_out` | C(12) | orchestration + fault isolation | Per-subscriber fan-out loop with four independent concerns: filter match, replay-watermark dedup (skip + clear-on-first-live), pre-ack buffer gate (queue + bounds + overflow-into-drop), and live send + lag-count drop. Each concern has a distinct per-subscriber side effect (cursor advance vs buffer append vs sendall vs drop) and metrics emission; collapsing them through helpers would force a per-iteration outcome tuple that re-dispatches inside the loop, breaking the compute→record→advance locality the existing shape gives. The pre-ack gate is one of the four arms. |
| `broadcast.py::Broadcast._read_subscribe` | C(17) | orchestration | Subscribe handshake orchestrator: parse + validate envelope → proto-version reject → optional token reject → field-validator dispatch → register subscriber → reserve pre-ack buffer → compute caught_up_at → emit subscribe_ack → optionally replay (guarded against a mid-replay `sqlite3.Error` → silent close, and an in-replay lag eviction → early return) → drain pre-ack buffer → clear buffer. Each stage has a distinct externally-observable outcome (silent EOF on shape error, typed `subscribe_rejected` frame on auth/version failure, full success path) and must run in strict sequence (the buffer reservation has to precede registration; the ack must precede replay; the drain must follow replay). Splitting into sub-orchestrators creates single-use indirections that hurt readability and obscure the ordering invariant this handshake encodes. |
| `_broadcast_sub.py::await_predicate` | C(16) | fault isolation | Independent terminal/skip arms: timeout / framing-EOF / clean-EOF / undecodable-skip / control-frame-drain (`daemon_heartbeat` + `subscribe_ack` are consumed off the wire but never handed to `decide` or advanced onto the cursor) / subscribe-rejected → typed exception via the `_REJECT_REASON_EXCEPTIONS` map / match / SIGINT. Each is a distinct externally-observable contract; decomposing forces tuple-return + re-dispatch and breaks compute→tag locality. |
| `sources/docker_watch.py::watch` | C(14) | fault isolation | Container event stream loop: connect → stream → reconnect/error arms. The capped backoff, event-epoch precedence and exit-code coercion are extracted to single-purpose helpers; the residual in-function arms are the fault-isolation core (stream-yielded / reconnect / unrecoverable) whose per-arm recovery contract does not decompose further without re-coupling them through a tuple-return. The stop-flag checks at the loop top and in the drop handler are per-arm shutdown contracts of the same fault-isolation class. |
| `sources/docker_watch.py::_iter_event_lines` | C(11) | fault isolation | Chunked /events reader: connect, register-with-stopper, per-line yield, and the generator finally that owns the connection close. The branches are the stop-seam contract (stopper registered around the live connection so a cross-thread shutdown wakes the blocked read; the reading thread alone closes) plus the HTTP-status and short-read arms — each a distinct recovery path that consolidating would re-entangle with the watch loop's own arms. |
| `_terminal.py::entity_key` | C(12) | type dispatch | Per-source / per-event-type dispatch (github/workflow_run, github/workflow_job, alertmanager/prometheus_alert, plus the pass-through default for sources without a stable upstream identity) with `type(v) is int and v > 0` domain narrowing on each GitHub id column and `isinstance(str)` plus non-empty narrowing on the alertmanager fingerprint. Each branch encodes a per-source identity contract that the coalesced consumer collapses on; collapsing them through one generic accessor would lose the per-source error semantics (e.g. silently coercing `bool` to int-stringified `True`/`False` collapse keys). |
| `cli/_shared.py::_render_table` | C(12) | type/value dispatch | Fixed-width aligned-table renderer shared by ``waitbus source list`` and ``waitbus allowlist list``. Per-column width compute, per-row stringify, header bolding when colour is enabled, optional per-row colour callback (plugin rows in ``source list`` go CYAN; allowlist rows are uncoloured). Each branch handles one presentation concern; splitting would force the caller to pass a pre-bolded header AND a pre-formatted row sequence, losing the centralisation that closes the previous duplication. |
| `cli/sources.py::source_show` | C(13) | orchestration | The 12-field deep-dive verb for one source: registry lookup, plugin-vs-built-in branch, optional PEP 740 attestation JSON read off disk (via the public `read_attestation_json` helper), TOFU pin lookup, optional `--json` format, payload-schema fully-qualified-name render. Each branch contributes one operator-visible field; per-field helpers would be single-use. The same `source list` row helper is reused for the table cells. |
| `sources/_attestation.py::verify_distribution` | C(11) | fault isolation + type dispatch | PEP 740 verification flow: locate the dist-info, read the PEP 610 `direct_url.json` for the wheel digest (graceful None on pip-name installs that don't write it), iterate `Provenance.attestation_bundles`, call `Attestation.verify` against each bundle's publisher identity. Each arm has a distinct fall-soft / fall-hard contract -- missing dist-info / missing direct_url.json / missing provenance file gracefully return None so the daemon can continue; sigstore `VerificationError` / `ConversionError` / malformed envelopes raise `AttestationVerificationError` so the registry's TOFU policy treats the plugin as untrusted. Splitting the arms into helpers loses the per-arm log-then-continue + raise-then-stop discrimination that the surrounding `_verify_publisher` relies on. |
| `sources/_protocol.py::SourceSpec.__post_init__` | C(12) | type/value dispatch | Per-field validation across the four `SourceSpec` fields (`name` regex, `event_types` non-empty + per-member regex, `payload_schema` isinstance, `api_version` positive-non-bool). Each branch raises a field-specific `ValueError` so plugin authors see which field failed; collapsing into a single error path would lose the per-field message and force authors to debug by inspection. The msgspec.Struct construction path requires this complexity because plugin authors are not in waitbus's type-check scope, so the runtime check is the only enforcement seam. |
| `sources/_config.py::load_plugin_policy` | C(12) | fault isolation | Three independent sources merged: `config.toml [plugins]` table (with per-field type-narrow guards on `autoload` / `allow` / `deny`), the `WAITBUS_DISABLE_SOURCE_AUTOLOAD` env override, and the `WAITBUS_PLUGINS` env list. The TOML parse + missing-file fallback + each guard arm produces a typed `PluginPolicy` regardless of which inputs are present; the per-field guards exist because TOML parse can yield wrong-type values when a hand-edited file is malformed. Splitting forces tuple-return through three private helpers that re-merge -- worse than the local accumulation. |
